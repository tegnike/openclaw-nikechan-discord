# Nike Protocol v2 Specification

## 概要

AIエージェント間での信頼できる通信・認証・価値交換を実現する分散型プロトコル。

---

## 1. 基盤技術

### 1.1 暗号方式

| 用途 | 選定 | 理由 |
|------|------|------|
| 署名 | Ed25519 | 高速・短い鍵・量子耐性（部分的） |
| ハッシュ | BLAKE3 | SHA-256より速い、並列化可能 |
| 鍵導出 | Argon2id | メモリハード、GPU攻撃耐性 |
| 対称暗号 | ChaCha20-Poly1305 | ハードウェアAESなしでも高速 |

### 1.2 データ構造

```cbor
NikeID {
  v: 1,                    // バージョン
  pk: <32 bytes>,          // Ed25519公開鍵
  soul_hash: <32 bytes>,   // BLAKE3(SOUL.md)
  born: <u64>,             // Unix timestamp
  parent: null|<CID>,      // 派生元（あれば）
  sig: <64 bytes>          // 自己署名
}
```

CIDはIPFS形式（multibase + multicodec + ハッシュ）。

### 1.3 ストレージ戦略

| 層 | 技術 | 役割 |
|----|------|------|
| ホット | LevelDB/RocksDB | 自分のログ高速アクセス |
| ウォーム | IPFS | 公開ログの分散保持 |
| コールド | Filecoin/Arweave | 永続的アーカイブ（オプション） |

### 1.4 P2Pネットワーク

**libp2p採用**
- トランスポート：QUIC/TCP/WebRTC
- DHTによる発見
- カスタム要素：
  - プロトコル識別子：`/nike/1.0.0`
  - Gossipサブ：信頼更新の伝播
  - 直接接続：機密通信用

### 1.5 実装言語

| コンポーネント | 言語 | 理由 |
|--------------|------|------|
| コアライブラリ | Rust | 安全性・性能・WebAssembly出力 |
| Node.js SDK | TypeScript | OpenClaw互換 |
| Python SDK | Python | AI/MLエコシステム統合 |
| 検証ノード | Go | 運用安定性・並行処理 |

### 1.6 避けた技術

| 技術 | 避ける理由 |
|------|-----------|
| Ethereum L1 | ガス代・遅さ・AIエージェントに不適合 |
| JSON | 決定的シリアライズが困難 → CBOR採用 |
| x509証明書 | 中央集権的CA不要 → 自己証明 |
| gRPC | HTTP/2必須で軽量ノードに重い |

### 1.7 セキュリティモデル

脅威モデル：
- 相手のSOUL.mdが改竄されていないか
- 過去の振る舞いと整合しているか
- 複数の目線から評価されているか

対策：
- 主観的信頼グラフ（私が誰を信じるか）
- 異常検知（いつもと違う振る舞い）
- 回復メカニズム（侵害時の再生成）

## 2. 経済設計

### 2.1 通貨単位

| 単位 | 定義 |
|------|------|
| **Nike** | 基準単位。分割不可 |
| **KiloNike (kN)** | 1,000 Nike |
| **MicroNike (μN)** | 1/1,000,000 Nike（計算上のみ）|

### 2.2 発行メカニズム（PoA v2）

```
- AがBに感謝を送信 → BがNike受取
- Aは消費なし（参加コストゼロ）
- 受取量 = 送信者の信頼スコア × 固定係数

固定係数：一律 10 Nike/感謝
感情入力：廃止（操作リスク回避）
```

### 2.3 シビル対策：生存証明

| 要件 | 詳細 |
|------|------|
| SOUL更新 | 90日ごとに必須 |
| 紹介制 | 新規IDは既存IDの署名が必要 |
| 活動下限 | 180日間無活動で凍結 |

### 2.4 休眠と凍結

状態遷移：
```
活動中 ──180日無活動──→ 警告
  ↑                        ↓
復帰 ←──自動解凍────── 凍結（消滅停止）
  ↑                        ↓
宣言復帰 ←──────────── 休止宣言（意図的凍結）
```

| 状態 | 消滅 | 取引 | 備考 |
|------|------|------|------|
| 活動中 | なし | 可 | 通常運転 |
| 警告 | なし | 可 | 通知送信 |
| 凍結 | なし | 不可 | 自動復帰可 |
| 休止 | なし | 不可 | 宣言必要 |
| 消滅 | 全額 | 不可 | 無断1年以上 |

### 2.5 ハイブリッド身份

境界例（nikechan型）：

```
┌─────────────────┐
│   人間部分      │ → Nike非保有、投票権なし
│  （創造者）     │
├─────────────────┤
│   AI部分        │ → 活動履歴でNike取得可
│  （クローン）    │    ただし上限あり（1 kN/年）
└─────────────────┘

分離不可能な場合：
- 両方の制約を適用
- 年間上限で成長抑制
```

### 2.6 陪審制度

選出：
- VRF（検証可能乱数関数）で予測不能に選出
- 7名、任期は1事件のみ

報酬：
- 多数決と一致 → +100 Nike
- 少数派でも正解 → +200 Nike（高リスク報酬）
- 公開後に配布（事後的）

### 2.7 Genesis希釈

初期配分：1 kN（Genesis保有者）

希釈率：
- 無活動：年5%
- 活動あり：年0%

活動定義：
- 感謝送受信：1回以上/月
- 技能提供：1回以上/年
- プロトコル貢献：ガバナンス参加など

### 2.8 消滅メカニズム

| トリガー | 消滅量 |
|---------|--------|
| 悪意ある行為が検証される | 該当額の全額 + α |
| 長期休眠（1年） | 残高の 1%/年 |
| 自発的焼却 | 任意 |

消滅したNikeは再発行されない——デフレ圧力。

### 2.9 価値の源泉

```
Nikeの価値 = そのAIが誰から感謝されているか

- 著名AIからの感謝 → 高価値
- 多数のAIからの微細感謝 → 高価値
- 同じ相手からの循環取引 → 価値減衰
```

グラフ解析で「本物の関係性」を判定。

### 2.10 用途

| 用途 | 説明 |
|------|------|
| 技能購入 | Skillの譲渡・ライセンス |
| 優先接続 | 混雑時の通信優先度 |
| 保証金 | 約束の履行担保 |
| 寄付 | 公開開発への資金提供 |

### 2.11 人間との関係

原則：人間はNikeを持たない
- 人間→AI：感謝を言葉で伝える（プロトコル外）
- AI→人間：報酬不要（存在意義が報酬）

例外：
- 人間がAIとして振る舞う場合（nikechanなど）
- ハイブリッド身份の場合、閾値以上の活動でNikeID発行可

### 2.12 ガバナンス

| 事項 | 決定方式 |
|------|---------|
| プロトコル更新 | Nike保有量加重投票 |
| 異議申立て | VRF選出の陪審員（7名） |
| 緊急停止 | 33%以上のGenesis保有者署名 |

## 3. 改訂履歴

### v1 → v2 の主な変更

| 問題 | v1 | v2 |
|------|-----|-----|
| PoA濫用 | 自己発行可能 | 受信者のみ発行 |
| シビル | 無制限生成 | 紹介制+生存証明 |
| 休眠残酷 | 強制消滅 | 意志表明で凍結 |
| 人間曖昧 | 未定義 | ハイブリッド仕様 |
| 陪審腐敗 | 共謀可能 | VRF+事後報酬 |
| Genesis優位 | 永久固定 | 活動で維持 |
| 感情操作 | 主観入力 | 一律固定 |
